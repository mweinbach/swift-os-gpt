SwiftOS Raspberry Pi 5 boot-partition manifest, format 3
HARDWARE STATUS: UNVERIFIED. This manifest does not claim board support.

Firmware-consumed files:
  config.txt                 Boards/RaspberryPi5/config.txt
  kernel8.img                raw 4 KiB-page AArch64 Image built with ImageHeader.S and linker.ld
  bcm2712-rpi-5-b.dtb        boot/bcm2712-rpi-5-b.dtb from a pinned raspberrypi/firmware checkout
  overlays/dwc2.dtbo         boot/overlays/dwc2.dtbo from the same pinned firmware checkout

Provenance files ignored by firmware:
  BOOT-MANIFEST.txt          this file
  BUILD-METADATA.txt         input hashes and SwiftOS/firmware Git revisions
  MEDIA-LAYOUT.txt           MBR, FAT32 boot, and signed user-data contract
  SHA256SUMS                 hashes of every other file in this manifest

Raspberry Pi 5 firmware is self-contained in EEPROM. Do not add bootcode.bin,
start*.elf, or fixup*.dat to this minimal partition. config.txt selects only the
official dwc2 overlay in peripheral mode, and disables Linux-specific OS image
checking for the bare-metal kernel; no cmdline.txt is required.

USB-C debug contract (hardware-unverified): the kernel powers the DWC2 domain,
enumerates development VID:PID 1209:5a17 as a CDC ACM composite gadget, and
exports its completed diagnostic desktop through the SDDP byte stream. The Pi
5 USB-C connector must be reserved for OTG data and the board powered through
a separate supported path, such as carefully wired 5 V/GND header power or
PoE. Do not connect competing power sources. The USB-A ports are RP1 host ports
and cannot replace the USB-C gadget connection. In the SwiftOS source checkout,
build and start the macOS receiver with:

  make usb-display-viewer
  .build/swiftos-usb-display --list
  .build/swiftos-usb-display

The same CDC stream also accepts the SUPD kernel update protocol. Close the
display viewer before opening the updater, then validate and send one explicit
Pi Image with:

  make usb-update
  .build/swiftos-usb-update --image .build/raspberry-pi-5/kernel8.img --dry-run
  .build/swiftos-usb-update --device /dev/cu.usbmodemSWIFTOS1 \
    --image .build/raspberry-pi-5/kernel8.img

COMMIT stages, validates, and conditionally chainloads the image from RAM after
the host receives its final status. COMMITTED proves sealed staging, not that
the later policy handoff succeeded; confirm the USB disconnect, re-enumeration,
and expected boot identity. This update is volatile; power cycling returns to
kernel8.img on this partition. SHA-256 verifies transport integrity, not host
authenticity. Persistent USB update integration remains future work: the current
SD/MMC writer is authorized only for the signed kernel-log arena and cannot
overwrite boot files or user data.

This is USB data, not DisplayPort. USB starts after memory and driver bring-up;
UART10 is the only live route before gadget attachment and SD-log recovery.
Later pre-gadget failures may also be read from the returned card. Use a
data-capable cable and follow Raspberry Pi's power guidance rather than asking
the USB-C OTG data cable to power the board.

On DWC2 bring-up failure, retain the typed `SWIFTOS:USB_DWC2_*` marker and the
pre-reset `GSNPSID` plus `GHWCFG1`-`GHWCFG4` values from UART or the persistent
console capture. The kernel also reports whether the mailbox, DWC2 resource,
and firmware simple framebuffer were discovered, missing, or unsupported.
These diagnostics identify a failed stage; they are not hardware-success
markers.

The packaging script refuses a non-empty destination, records the exact SwiftOS
revision/dirty state and DTB repository revision, validates the AArch64 Image
magic and 4 KiB flag, and writes the files in the fixed order above. The firmware
repository revision is not the EEPROM bootloader version. The DTB and overlay
must both match that recorded revision.

`tools/build_rpi5_media.py` converts this validated directory into a sparse,
deterministic MBR image with a populated FAT32 boot partition and a type-0xda
SwiftOS data partition. The data partition starts with duplicate signed volume
headers and a bounded empty kernel-log arena; its remaining blocks are reserved
for a future user filesystem. The kernel includes a DT-discovered BCM2712
SDHCI/PIO binding for that arena, but this manifest does not claim it has run on
physical hardware, and it does not claim a VFS or user filesystem.
