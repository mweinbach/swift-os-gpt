# Raspberry Pi 5 8 GB boot evidence — `fe25ec7`

This note records the third returned-card trace from the physical Raspberry Pi
5 8 GB used on 2026-07-20. It proves another complete durable boot, advances the
USB failure from firmware power policy into live DWC2 gadget service, and
narrows the Ethernet failure to the RP1 clock aperture. It is not a claim of
complete Raspberry Pi 5 support.

## Physically observed working path

The artifact again reached EL1, parsed the firmware FDT, classified about 8 GB
of memory, installed final page tables and exception vectors, initialized
GICv2, and received timer interrupts. PSCI brought CPUs 1, 2, and 3 online;
each secondary completed both scheduled work items on its own stack with local
timer-interrupt evidence before `SMP_WORK_OK` and `SMP_OK`.

The removable SD path reported the exact 250,347,520-block geometry, validated
the MBR and signed type-`0xda` data volume, recovered the previous log epoch,
remounted SwiftFS, and flushed the third epoch. The diagnostic renderer and
animation path completed against its RAM-backed surface.

## Exact current blockers

### USB-C debug, display, and update

The kernel translated the DT-described controller to
`USB_DWC2_BASE=0x0000001000480000`. The revised Pi policy accepted the
firmware's well-formed legacy-device-off response as an unmanaged handoff and
continued. The following ordered markers were retained:

```text
SWIFTOS:USB_POWER_UNMANAGED_OFF
SWIFTOS:USB_DEBUG_ATTACHED
SWIFTOS:USB_UPDATE_READY
...
SWIFTOS:USB_DEBUG_FAULT
```

This proves DWC2 identity and capability checks, AHB-idle/reset, device-mode
selection, FIFO and power programming, software connect, and update-service
attachment. `USB_DEBUG_ATTACHED` is a software activation marker; it is not
host-enumeration evidence. No `USB_DEBUG_CONFIGURED` or frame marker followed,
and a contemporaneous macOS IORegistry/USB probe observed no SwiftOS USB
descriptor or CDC device. The failure is therefore in the live gadget service
before Chapter 9 configuration, not in the firmware mailbox handoff and not
evidence of a missing macOS class driver.

The retained artifact does not distinguish bus reset, enumeration speed, or a
typed service fault because those events were not yet persisted. Inspection of
the bounded EP0 state machine found that it rejected a valid DWC2 ordering in
which endpoint-zero OUT completion precedes setup-done. That is the primary
next-artifact hypothesis, not a claim proven by this trace.

### RP1 Ethernet

The atomic clock-enable artifact retained:

```text
SWIFTOS:RP1_NET_STARTING
SWIFTOS:RP1_NET_PREPARING
SWIFTOS:RP1_NET_BOARD_STAGE=0x0000000000000002
SWIFTOS:RP1_NET_BOARD_REGISTER=0x0000001f00018014
SWIFTOS:RP1_NET_BOARD_EXPECTED=0x0000000000000800
SWIFTOS:RP1_NET_BOARD_OBSERVED=0x0000000000000002
SWIFTOS:RP1_NET_BOARD_FAILED
```

Stage 2 is the RP1 SYS clock. The normal control register returned the plausible
source-selection value `0x2`, but its enable bit 11 was not visible on the one
immediate read after writing the atomic SET alias at `0x1f0001a014`.
Ethernet/TSU clock setup, PHY reset, GEM, MDIO, autonegotiation, link, and DHCP
were consequently never attempted. Router absence is consistent with this
earlier failure. The trace does not yet distinguish a posted-write delay, a
normal-register fallback requirement, an uninitialized parent clock, or a
deeper RP1 PCIe/BAR ownership problem.

The next artifact must retain pre/write/post clock state, a posted-write drain,
bounded readback polling, and the result of a same-address read/modify/write
fallback before moving to read-only PCIe/BAR diagnostics.

### HDMI and GPU

`PI_SIMPLE_FB_MISSING` was retained before `GRAPHICS_DIAGNOSTIC`. The completed
RAM compositor and animation markers do not prove HDMI scanout, EDID, selected
mode, PPI, refresh timing, HVS presentation, or V3D command submission. No HDMI
or native-GPU claim follows from this capture.

## Still unproven on physical hardware

USB enumeration and live update, RP1 Ethernet link/DHCP, HDMI output, V3D GPU
submission, keyboard and mouse input, and EL0 cross-core process migration all
remain unverified on the Pi. QEMU and host-test coverage for those paths must
not be reported as Raspberry Pi hardware support.
