# Raspberry Pi 5 8 GB boot evidence — `64dae29`

This note records the fourth returned-card trace from the physical Raspberry Pi
5 8 GB used on 2026-07-20. It proves another complete durable boot, records a
DWC2 bus reset and high-speed enumeration-done event for the first time, and
isolates the Ethernet failure to a normal/atomic-alias clock pre-read. It is not
a claim of complete Raspberry Pi 5 support.

- Data superblock status: healthy
- Persistent records: 3,314, sequence 1 through 3,314
- Persistent gaps or resets: none
- Kernel boot epochs: four
- Epoch record counts: 642, 723, 729, and 1,220
- Newest epoch: persistent sequence 2,095 through 3,314
- Newest console: 4,125 bytes with complete message boundaries and valid UTF-8
- Firmware-provided device-tree address in every epoch: `0x03000000`
- Architectural counter frequency in every epoch: 54,000,000 Hz

The newest boot reported `STORAGE_LOG_RECOVERY_READY=0x82e`, exactly the prior
epoch's last persistent sequence, followed by `STORAGE_LOG_FLUSH_OK=0x82f`.
It also reported `SWIFTFS_REMOUNTED` and `SWIFTFS_READY`, then continued
appending through persistent sequence 3,314. This is the fourth consecutive
gap-free power-cycle epoch and the third remount after the initial format.

The aggregate console is labelled incomplete only because it intentionally
crosses four kernel epochs. The newest epoch is independently complete from
`SWIFTOS:BOOT` through its final full log-flush marker. Completeness means that
the retained epoch was reconstructed without a prefix/interior gap or partial
message; it does not mean USB or Ethernet succeeded, nor prove that no later
volatile event occurred after the final durable record.

## Physically observed working path

The artifact reached EL1, parsed the firmware FDT, classified roughly 8 GB of
memory, installed final page tables and exception vectors, initialized GICv2,
and received timer interrupts. PSCI brought CPUs 1, 2, and 3 online; each
secondary completed its scheduled work and processor-local timer proof before
`SMP_WORK_OK` and `SMP_OK`.

The SD path reported the exact 250,347,520-block geometry, validated the MBR and
signed type-`0xda` data volume, recovered the prior log endpoint, remounted
SwiftFS, and flushed the fourth epoch. The headless compositor and bounded
animation proof completed against the kernel-owned diagnostic surface.

## Exact current blockers

### USB-C debug, display, and update

The retained order includes:

```text
SWIFTOS:USB_POWER_UNMANAGED_OFF
SWIFTOS:USB_DEBUG_ATTACHED
SWIFTOS:USB_UPDATE_READY
...
SWIFTOS:USB_DEBUG_BUS_RESET
SWIFTOS:USB_DEBUG_ENUMERATED_SPEED=0x0000000000000000
...
SWIFTOS:USB_DEBUG_FAULT_REASON=0x0000000000000006
SWIFTOS:USB_DEBUG_FAULT_GADGET_STATE=0x0000000000000000
SWIFTOS:USB_DEBUG_FAULT_CONTROLLER_STATE=0x0000000000000002
SWIFTOS:USB_DEBUG_FAULT_GLOBAL=0x0000000000000010
SWIFTOS:USB_DEBUG_FAULT_ENDPOINT=0x0000000000000000
SWIFTOS:USB_DEBUG_FAULT_BUS_SPEED=0x0000000000000000
SWIFTOS:USB_DEBUG_FAULT_RX_STATUS=0x00000000000c0080
SWIFTOS:USB_DEBUG_FAULT
```

This physically proves DWC2 attachment, a USB bus reset, and enumeration-done
at high speed (`0x0`). It does not prove a successful device-descriptor exchange,
Chapter 9 configuration, CDC creation, display frame, or live update. Neither
`USB_DEBUG_CONFIGURED` nor `USB_DEBUG_FRAME` was retained.

Receive status `0x000c0080` decodes as endpoint zero, eight bytes, data PID
zero, packet status 6 (`SETUPRX`), and frame zero. It is a structurally valid
eight-byte setup packet, not malformed FIFO status or bulk traffic. With fixed
512-byte receive and 64-byte EP0 staging buffers, the artifact's remaining
terminal predicate was a new `SETUPRX` arriving while an earlier setup remained
staged. The pre-fault gadget state had returned to attached after the one-shot
high-speed marker, which also indicates a later reset that the one-shot marker
did not repeat. The long synchronous RP1 preparation window plausibly allowed
receive-status backlog to accumulate; that timing explanation is an inference,
not a separately measured latency claim.

Current source accepts a valid newer setup as an abort of the older EP0 control
transfer and drains at most eight RX-status entries per cooperative pass. Host
tests reproduce `SETUPRX(A)`, optional `OUTDONE`, `SETUPRX(B)`, `SETUPDONE` and
prove only B executes, while a ninth entry remains for the next pass. This fix
was made after `64dae29` and still requires another physical boot.

### RP1 Ethernet

The retained SYS/PLL and clock-attempt block includes:

```text
SWIFTOS:RP1_NET_CLOCK_SYS_CTRL=0x0000000000000002
SWIFTOS:RP1_NET_CLOCK_SYS_DIV_INT=0x0000000000000001
SWIFTOS:RP1_NET_CLOCK_SYS_SEL=0x0000000000000004
SWIFTOS:RP1_NET_CLOCK_PLL_SYS_CS=0x0000000080000001
SWIFTOS:RP1_NET_CLOCK_PLL_SYS_PWR=0x0000000000000004
SWIFTOS:RP1_NET_CLOCK_PLL_SYS_PRIM=0x0000000000051010
SWIFTOS:RP1_NET_CLOCK_PLL_SYS_SEC=0x0000000080000800
SWIFTOS:RP1_NET_CLOCK_STAGE=0x0000000000000002
SWIFTOS:RP1_NET_CLOCK_METHOD=0x0000000000000000
SWIFTOS:RP1_NET_CLOCK_RESULT=0x0000000000000002
SWIFTOS:RP1_NET_CLOCK_INITIAL=0x0000000000000002
SWIFTOS:RP1_NET_CLOCK_ALIAS_INITIAL=0x0000000000000000
SWIFTOS:RP1_NET_CLOCK_ALIAS_DRAIN=0x0000000000000000
SWIFTOS:RP1_NET_CLOCK_FINAL=0x0000000000000002
SWIFTOS:RP1_NET_CLOCK_POLLS=0x0000000000000000
SWIFTOS:RP1_NET_CLOCK_ELAPSED_TICKS=0x0000000000000000
SWIFTOS:RP1_NET_BOARD_STAGE=0x0000000000000002
SWIFTOS:RP1_NET_BOARD_REGISTER=0x0000001f00018014
SWIFTOS:RP1_NET_BOARD_EXPECTED=0x0000000000000002
SWIFTOS:RP1_NET_BOARD_OBSERVED=0x0000000000000000
SWIFTOS:RP1_NET_BOARD_FAILED
```

The normal SYS clock control register read `0x2`, its SET-alias pre-read returned
zero, and the artifact treated that inequality as a terminal invariant before
performing any write or poll. The board expected/observed pair therefore
compares those two pre-read values; it does not say the desired enable value was
`0x2`. Enable bit 11 (`0x800`) remained unset. Ethernet/TSU clocks, PHY reset,
GEM, MDIO, autonegotiation, link, and DHCP were never attempted.

The RP1 specification describes `clocks_main` as atomic-capable, but the
physical access path did not provide the assumed mirrored alias read. Raspberry
Pi's published Linux clock driver enables a leaf through a locked normal CTRL
read/modify/write. Current SwiftOS consequently treats alias reads as telemetry:
if normal CTRL is already enabled it performs no write; if the alias diverges it
skips the suspect alias and uses a fresh, bounded normal-register RMW; if the
alias agrees it retains the atomic SET path. This fix was made after `64dae29`
and is not yet physical Ethernet evidence.

### HDMI and GPU

`PI_SIMPLE_FB_MISSING` was retained before `GRAPHICS_DIAGNOSTIC`. The completed
RAM compositor and animation markers do not prove HDMI scanout, EDID, selected
mode, PPI, refresh timing, HVS presentation, or V3D command submission.

## Still unproven on physical hardware

USB descriptor/configuration and live update, RP1 Ethernet link/DHCP, HDMI
output, V3D GPU submission, keyboard and mouse input, and EL0 cross-core process
migration remain unverified on the Pi. QEMU and host-test coverage for those
paths must not be reported as Raspberry Pi hardware support.
