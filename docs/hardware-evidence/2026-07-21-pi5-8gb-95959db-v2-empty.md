# Raspberry Pi 5 8 GB v2 boot attempt — `95959db`

This note records the first returned-card inspection of the format-v2 A/B
media on the physical Raspberry Pi 5 8 GB. The card remained internally
healthy, but it contained no persistent boot epoch. This is failure evidence,
not proof that the firmware loaded SwiftOS or that the v2 boot path works.

Fresh macOS discovery identified the removable card as `/dev/disk7` for this
inspection. The unrelated 2 TB Echo SSD remained `/dev/disk6` and was not
opened. Device numbers are observations from this capture only and must never
be reused for a later media operation.

The returned MBR retained the complete v2 geometry:

| Entry | Start block | Block count | Type | Role |
| --- | ---: | ---: | ---: | --- |
| 1 | 1 | 2,047 | `0x01` | FAT12 selector/rescue |
| 2 | 2,048 | 262,144 | `0x0c` | payload A |
| 3 | 264,192 | 262,144 | `0x0c` | payload B |
| 4 | 526,336 | 249,821,184 | `0xda` | journal, logs, and SwiftFS |

The selector's `autoboot.txt` still selected partition 2 normally and
partition 3 under tryboot. Its rescue files matched the packaged digests.
Every `SHA256SUMS` entry passed independently on both read-only mounted
payload partitions, and both named the exact clean revision above. This rules
out an incomplete whole-card write or corrupted payload file as the observed
failure.

## Returned persistent-log result

The read-only inspector reported:

```text
source: /dev/rdisk7 (read-only)
data superblocks: healthy
capture: empty; boot epochs 0
sequences: empty
console: 0 bytes; complete no
diagnostic markers: none
console: no retained console bytes
```

Both `SWOSDATA` superblocks validated and still described the intended 4,096
block log arena plus the disjoint SwiftFS range. There were zero persistent
records, zero `BOOT` records, and zero retained console chunks. Therefore this
attempt never completed a valid persistent-log append or synchronization.

That result does **not** prove that EEPROM firmware never loaded
`kernel8.img`. Early console bytes remain in RAM until the BCM2712 SDHCI path,
MBR/data-volume discovery, and persistent-log service are ready. Firmware
rejection, an early kernel fault, watchdog reset, or SD initialization failure
can all leave the same pristine arena.

## Contemporaneous USB observation

The macOS control probe watched through a 90-second boot window and remained
at `stage: absent`. IOKit contained no SwiftOS `1209:5a17` device, no associated
CDC node appeared under `/dev/cu.usbmodem*`, and recent host logs contained no
SwiftOS USB attach event. This proves that no complete SwiftOS USB descriptor
reached the Mac; it does not distinguish an unpowered or mis-cabled board from
a pre-descriptor guest failure. The exact power and cable topology was not
independently captured for this attempt.

## Regression boundary

The last physically successful format-v1 payload used a FAT32 boot partition
at block 2,048 whose FAT `BPB_HiddSec` value was also 2,048. The v2 builder
instead wrote zero into `BPB_HiddSec` for payload A at block 2,048 and payload
B at block 264,192 so their raw extents could be byte-identical. Although host
parsers accept that choice, it is noncanonical partition metadata and is a
concrete firmware-compatibility regression relative to every successful boot.

The same v2 artifact also enabled and adopted the BCM2712 PM watchdog before
storage logging on every boot. That MMIO path had host-model coverage but no
physical execution evidence. A slot rejection, a watchdog MMIO failure, or an
early watchdog reset can therefore all precede the first durable record.

Raspberry Pi's published boot contract explicitly permits FAT12, FAT16, and
FAT32 and the selector policy matches its documented A/B `autoboot.txt`
example. FAT12 alone is not an evidenced failure. The installed EEPROM build,
`BOOT_ORDER`, tryboot capabilities, and `PARTITION_WALK` setting remain
unknown; rescue fallback must not be claimed without capturing them.

## Next acceptance boundary

The next physical artifact should restore each payload partition's actual
hidden-sector value and avoid touching the unverified PM watchdog on an
ordinary stable boot. Candidate-only watchdog probation must remain fail
closed. Before calling v2 bootable, physical evidence must record at least one
durable boot epoch, the firmware-reported boot partition, and either a complete
USB configuration or a typed USB fault. UART10 or HDMI bootloader diagnostics
are still required to capture the EEPROM version and any failure before kernel
entry.
