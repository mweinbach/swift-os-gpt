# Raspberry Pi 5 8 GB boot evidence — `a0bcee7`

This note records the second returned-card trace from the physical Raspberry Pi
5 8 GB used on 2026-07-20. It narrows the USB and Ethernet failures and proves a
power-cycle recovery of the first boot's persistent records. It is not a claim
of complete Raspberry Pi 5 support.

The FAT32 boot volume matched all seven entries in the packaged
`SHA256SUMS` before extraction. The whole card was then unmounted and opened
read-only by `tools/swiftos_pi_logs.py`. Its fresh host discovery selected
`/dev/rdisk4`; the unrelated fixed 2 TB `/dev/disk6` device was not opened.

## Persistent-log integrity and durability

- Data superblock status: healthy
- Persistent records: 1,365, sequence 1 through 1,365
- Persistent gaps or resets: none
- Kernel boot epochs: two
- First epoch: 642 contiguous kernel records
- Newest epoch: 723 contiguous kernel records, persistent sequence 643 through
  1,365
- Firmware-provided device-tree address in both epochs: `0x03000000`
- Architectural counter frequency in both epochs: 54,000,000 Hz

The newest boot reported `STORAGE_LOG_RECOVERY_READY=0x282`, exactly the prior
epoch's last persistent sequence, then `STORAGE_LOG_FLUSH_OK=0x283`. It also
reported `SWIFTFS_REMOUNTED` and `SWIFTFS_READY`. This proves that the second
boot recovered the first boot's durable arena, remounted the existing SwiftFS
user region rather than formatting it, and appended the new boot. The combined
canonical console is marked incomplete only because it crosses two kernel
epochs; its retained records have no sequence gap or malformed console chunk.

## Physically observed working path

The new artifact again reached EL1, parsed the FDT, classified about 8 GB of
memory, installed final page tables and exception vectors, initialized GICv2,
and received timer interrupts. PSCI brought CPUs 1, 2, and 3 online; each ran
both scheduled SMP work items on its own stack and recorded processor-local
timer interrupts.

The SD controller again reported the card's exact 250,347,520-block geometry,
validated the MBR and signed type-`0xda` data volume, recovered the previous log
epoch, remounted SwiftFS, and flushed the current epoch. The CPU compositor and
animation path also completed against its RAM-backed diagnostic surface.

## Exact current blockers

### USB-C debug and display

The kernel discovered the firmware mailbox and DT-described DWC2 controller,
then translated the controller to `USB_DWC2_BASE=0x0000001000480000`. The
firmware property transaction completed with a structurally valid power-state
response, but the response described the legacy USB device as present and off.
The Pi-specific policy emitted `USB_POWER_STATE_MISMATCH` and returned before
the first DWC2 MMIO probe.

There is consequently no physical evidence yet for `USB_DEBUG_ATTACHED`, host
enumeration, CDC configuration, a display frame, or live kernel update on this
artifact. A contemporaneous macOS probe found neither the expected
`1209:5a17` identity nor an associated CDC callout device. This is a guest
power-handoff blocker, not evidence of a missing macOS class driver.

### RP1 Ethernet

The kernel reached `RP1_NET_STARTING` and `RP1_NET_PREPARING`, then retained the
complete board-preparation diagnostic:

```text
SWIFTOS:RP1_NET_BOARD_STAGE=0x0000000000000002
SWIFTOS:RP1_NET_BOARD_REGISTER=0x0000001f00018014
SWIFTOS:RP1_NET_BOARD_EXPECTED=0x0000000000000800
SWIFTOS:RP1_NET_BOARD_OBSERVED=0x0000000000000002
SWIFTOS:RP1_NET_BOARD_FAILED
```

Stage 2 is the first RP1 system-clock control operation. The address is the
DT-translated RP1 main-clock aperture plus offset `0x14`; bit 11 did not latch
on readback. No PHY reset, MDIO, autonegotiation, link, DHCP, or packet path was
attempted afterward. The router's lack of a client or link is consistent with
this earlier board-preparation failure.

### HDMI and GPU

`PI_SIMPLE_FB_MISSING` was retained before the kernel selected
`GRAPHICS_DIAGNOSTIC`. `USB_DISPLAY_READY`, `FRAMEBUFFER_READY`, and the
animation markers prove only the CPU compositor's RAM surface. They do not
prove HDMI scanout, display timing, PPI, refresh metadata, HVS modesetting, or
V3D command submission.

## Still unproven on physical hardware

USB enumeration and live update, RP1 Ethernet link/DHCP, HDMI output, V3D GPU
submission, keyboard and mouse input, and EL0 cross-core process migration all
remain unverified on the Pi. Their QEMU or host-test coverage must not be
reported as Raspberry Pi hardware support.
