# Raspberry Pi 5 8 GB boot evidence — 2026-07-20

This note records the first returned-card evidence from a physical Raspberry Pi
5 8 GB. It is a hardware observation, not a declaration that the board is fully
supported.

## Artifact and media identity

- SwiftOS revision: `9e93dacd042cffb36fe118da0617e6ac83c9de64`
- Pinned `raspberrypi/firmware` file-set revision:
  `ed05d403048f2956d9d3653acd996157363e94fe`
- EEPROM bootloader build: not retained
- `kernel8.img` SHA-256:
  `6c741c347894661cc2b6371f4a6c6799cfa744b2567c256d2332aa1d1164c36a`
- Returned microSD geometry: 128,177,930,240 bytes, 250,347,520 logical
  512-byte blocks
- Boot partition: MBR partition 1, type `0x0c`, start block 2,048, 524,288
  blocks
- SwiftOS data partition: MBR partition 2, type `0xda`, start block 526,336,
  249,821,184 blocks

The mounted FAT32 partition matched every file named by the package's expected
`SHA256SUMS` hash manifest. Unrelated macOS metadata was present but was not
part of the hashed boot payload. The manifest authenticates no signer.

## Persistent-log integrity

The card was unmounted and inspected read-only through

count supplied as an independent bound. The raw JSON capture remains an ignored

`.build/hardware-captures/pi5-persistent-log.json`; it is intentionally not
committed.

- Data superblock status: healthy
- Persistent records: 642
- Boot epochs: one
- Persistent sequence: 1 through 642
- Missing records, gaps, or resets: none
- Canonical console stream: 2,543 bytes, complete, valid UTF-8
- Boot counter frequency: 54,000,000 Hz
- Firmware-provided device-tree address: `0x03000000`

## Physically observed working path

The persistent console proves that this artifact reached EL1, parsed the FDT,
discovered about 8 GB of usable memory, installed final page tables and
exceptions, initialized the interrupt controller, and received timer
interrupts. PSCI brought CPUs 1, 2, and 3 online. Each secondary CPU completed
two independently accounted work items on a distinct stack while recording
processor-local timer interrupts.

The retained secondary timer counts were CPU1 `0x22`, CPU2 `0x29`, and CPU3
`0x2f`. CPU0 separately emitted `TIMER_IRQ`, `TIMER_1`, `TIMER_2`, and
`TIMER_3` before `READY`.

The same boot initialized the SD controller, read the MBR and type-`0xda` data
partition, validated the SwiftOS data superblock, formatted and mounted the
SwiftFS user-data region, completed an empty-arena log scan
(`STORAGE_LOG_RECOVERY_READY=0`), and flushed the current boot back to the card.
This is first-persistence evidence, not a prior-boot recovery or power-cycle
durability result. The compositor and animation path also ran against the
headless USB-display render surface.

## Failed or unproven paths

- `SWIFTOS:PI_SIMPLE_FB_MISSING`: firmware did not hand off a usable
  simple-framebuffer. This does not prove native HDMI scanout or V3D/HVS
  acceleration, and the capture does not establish whether HDMI was attached.
- USB stopped at `SWIFTOS:USB_POWER_STATE` before DWC2 device initialization.
  `USB_DISPLAY_READY` only described the RAM-backed rendering surface. A
  contemporaneous host observation found no SwiftOS USB device on the Mac.
- RP1 Ethernet stopped after `RP1_NET_PREPARING` at
  `RP1_NET_BOARD_FAILED`, before a link or network stack could exist. A
  contemporaneous router observation found no client; that was not evidence
  that the kernel failed to boot.
- The physical trace contains no EL0 scheduling or migration markers. EL0
  cross-core migration remains QEMU-only evidence until a Pi trace proves it.
- Returned-card persistent records are not evidence of UART10 electrical
  output; no UART capture was retained for this boot.
- HDMI modesetting, native GPU command submission, USB enumeration, Ethernet
  link/DHCP, keyboard, and mouse remain hardware-unverified.

The next artifact should treat a well-formed firmware response that reports the
legacy USB power device as unavailable as an unmanaged handoff and continue to
DWC2 probing. It should also persist exact RP1 board-preparation stage,
register, expected-value, and observed-value markers so the next returned card
identifies the Ethernet failure rather than collapsing it into one generic
marker.
